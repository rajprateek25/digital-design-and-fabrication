# LAB 2:  Introduction to Arduino

## Sub-circuit 1: Connecting the buzzer

### Objective 
Construct a basic digital output circuit on a breadboard using an Arduino Uno, and to analyze how modifying the timing delays and `HIGH`/`LOW` state sequences impacts the component's output behavior.

---

### Components Utilized

* Microcontroller: Arduino Uno
* Output Component: Piezo Buzzer ($B_1$) 
* Current-Limiting Resistor: 220 Ω ($R_1$)
* Prototyping Hardware: Breadboard & Jumper Wires
* Power Source: USB connection (5V)

---

### Pin Mapping Table

| From Component / Pin | To Component / Pin | Purpose |
| --- | --- | --- |
| Arduino Digital Pin 12 | Resistor $R_1$ (220 $\Omega$) | Digital output signal delivery |
| Resistor $R_1$ (Terminal 2) | Buzzer B1 (Positive Terminal) | Current-limited signal input |
| Buzzer B1 (Negative Terminal) | Arduino GND | Completing the ground return path |

---

### Observations

After running the baseline `buzzer_test.ino` code and systematically modifying the delay durations and code sequences, the following behaviors were observed:

#### A. Baseline Test Condition

* **Observation:** When the digital pin cycles between `HIGH` and `LOW` with standard delays (e.g., 1000ms), the buzzer produces a distinct, intermittent **beeping sound**.
* **Technical Reason:** The `HIGH` state supplies 5V to the buzzer, activating the internal piezo element, while the `LOW` state drops the voltage to 0V, silencing it.

#### B. Adjusting Delay Durations

* **Shortened Delays (e.g., 50ms to 100ms):** The beep interval fast-forwards into a rapid, frantic clicking or ticking sound. If the delays are reduced to microseconds, the individual on/off states blur together, changing the perceived pitch or sounding like a continuous, scratchy buzz.
* **Lengthened Delays (e.g., 3000ms+):** The buzzer stays on or off for long stretches, shifting the output from a rhythmic notification sound to a sustained alarm state.

#### C. Altering the Sequence of HIGH/LOW States

* **Unequal Sequences (e.g., `HIGH` for 200ms, `LOW` for 1000ms):** Creating asymmetrical delays produces a distinct "heartbeat" or pulse-style notification (a quick blip followed by a long silence).
* **Consecutive Duplicate States:** Adding back-to-back `HIGH` statements without a `LOW` delay yields no acoustic change; the buzzer simply stays constantly active, proving that a state *transition* coupled with a time *delay* is mandatory to generate a rhythmic audio pattern.

---

### Conclusion

This exercise demonstrates how digital output signals from an Arduino can control auditory peripherals. By utilizing a 220 $\Omega$ resistor, the microcontroller's I/O pin was safely protected from overcurrent. The experiment highlights the critical relationship between software-defined timing delays and real-world physical feedback.

## Sub-circuit 2: Connecting the LCD screen

### Objective

Interface a $2 \times 16$ character display with an Arduino Uno using the $I^2C$ (Inter-Integrated Circuit) protocol, minimizing wire complexity to 4 lines, and to determine the device's hardware address for correct data transmission.

---

### Technical Context: Understanding $I^2C$

Unlike a standard parallel LCD connection which requires up to 10-12 data and control lines, the $I^2C$ protocol utilizes a shared-bus master/slave topology requiring only two signal wires:

* **SCL (Serial Clock):** The timing and synchronization signal generated by the master device (Arduino).
* **SDA (Serial Data):** The bidirectional line used to transmit data frames and device addresses.

---

### Pin Mapping Table

| From Arduino Uno Pin | To I2C LCD Backpack Pin | Function |
| --- | --- | --- |
| 5V | VCC | $+5\text{V}$ Power Supply |
| GND | GND | Common Ground Path |
| A4 / SDA | SDA | Serial Data Line (Data transfer) |
| A5 / SCL | SCL | Serial Clock Line (Bus timing) |

---

### Software Implementation & Experimental Methodology

### Step 1: Library Dependency Installation

Before testing, the external `<LiquidCrystal_I2C.h>` abstraction library was installed into the Arduino IDE environment. This library translates standard alphanumeric characters into binary command matrices native to the LCD controller.

### Step 2: Running the I2C Scanner (`I2C_scanner.ino`)

Because multiple peripherals can theoretically share the same SDA/SCL lines, the Arduino must target a specific hexadecimal memory address.

* **Action:** An address scanning utility script was uploaded to ping every device on the bus.
* **Observation:** The Serial Monitor output successfully discovered an active device. The hardware address returned was **`0x27`** (or **`0x20`**, depending on your specific board variant).

### Step 3: Display Testing (`LCD_test.ino`)

Using the discovered address, the primary test script was configured as follows:

```cpp
LiquidCrystal_I2C lcd(0x27, 16, 2); // Hex address, 16 columns, 2 rows

```

---

### Observations

* **Backlight and Initial Power State:** Upon initial wiring, the LCD screen lit up but showed white blocks on the first line. This verified that the power lines (VCC/GND) were working properly, and that the display controller was awaiting initialization commands over $I^2C$.
* **Successful Data Serialization:** Once `LCD_test.ino` was executed with the correct hex address, the blocks cleared, and smooth, stable alphanumeric characters appeared.
```
lcd.print("Hello! Prateek.");
```
* **Incorrect Address Behavior:** Accidentally changing the address inside the code (e.g., trying to write to `0x3F` instead of `0x27`) resulted in a compilation success but complete failure on the physical device. The screen did not update because the data packets on the SDA line were ignored by the display hardware.

---

### Conclusion

Interfacing via $I^2C$ drastically lowers the physical layout footprint on the breadboard, reducing potential wiring mistakes. This task underscored a crucial lesson in embedded systems engineering: hardware communication requires both correct physical wiring paths and exact software identification parameters (hexadecimal device addressing) to function properly.

## Sub-circuit  3: Expanding the Setup with a Real Time Clock
### Objective
Expand the microprocessing system by connecting a Real Time Clock module to the existing $I^2C$ hardware bus, verifying that multiple slave devices can seamlessly share a single set of data and clock lines without communication conflict.

---

### Technical Context: $I^2C$ Bus Sharing & Hardware Addressing

The primary advantage of the $I^2C$ protocol is its ability to support multiple peripheral devices wired in parallel across the exact same lines: **SDA** and **SCL**.

Because each device has a unique, hardcoded factory hexadecimal address, the Arduino master can isolate communications. When the master transmits a data packet, it prefixes it with the specific target device address; all other peripherals on the bus read the address, recognize it is not theirs, and remain quiet.

---

### Pin Mapping Table

| From Arduino Uno Pin | To I2C RTC Module Pin | Function | Bus Status |
| --- | --- | --- | --- |
| 5V | VCC | $+5\text{V}$ Power Input | Shared (Parallel) |
| GND | GND | Common Ground Path | Shared (Parallel) |
| A4 / SDA | SDA | Serial Data Line | Shared (Parallel Bus) |
| A5 / SCL | SCL | Serial Clock Line | Shared (Parallel Bus) |

> **Hardware Detail:** The RTC module features an independent onboard coin cell battery. This backup power source ensures the internal crystal oscillator continues counting time accurately even if the primary 5V line from the Arduino is completely disconnected.

---

### Software Implementation

### Step 1: Library Dependency Installation

To interface with the registers of the clock chip, the external `<RTClib.h>` library was installed and integrated into the software architecture.

### Step 2: Multi-Device Address Identification

To isolate the RTC from the LCD screen, the `I2C_scanner.ino` utility was executed with both devices actively plugged into the breadboard.

* **Observation:** The Serial Monitor output successfully detected **two distinct active addresses** simultaneously on the bus.
* **Address Mapping:**
* **`0x27`** (or `0x20`): Identified as the $2 \times 16$ LCD Screen.
* **`0x68`**: Successfully identified as the Real Time Clock module (the standard hardcoded address for DS1307 / DS3231 chips).

---

###  Observations

* **Parallel Bus Stability:** Despite splitting the data (SDA) and clock (SCL) lines to feed two different physical chips at the same time, signal integrity remained perfectly intact. The microcontroller suffered zero lag or data packet collisions.
* **Time Retention Test:** The `RTC_LCD_test.ino` script was successfully uploaded, bringing up a live clock display on the LCD. The USB cable was then pulled to completely power down the system for several minutes. Upon reconnection, the clock did not reset to zero; it immediately displayed the correct, continuously updated time, proving the functionality of the RTC backup cell battery.
* **Missing Device Error Handling:** Intentionally commenting out the initialization of the RTC address inside the script caused the software to freeze or report dummy values (like `00/00/0000 00:00:00`), highlighting that the software must explicitly declare and ping the specific `0x68` hex value to process incoming time registers.

---

### Conclusion

This exercise  validated the expansiveness of the $I^2C$ protocol. By utilizing parallel hardware bus architecture, we added full time-keeping capabilities to our circuit without sacrificial consumption of extra I/O pins on the Arduino Uno. The project underscores the value of hardware-based address isolation in complex embedded designs.

## Sub-circuit 4: Using the Push Button

### Objective
Expand the system with an asynchronous manual input device using an instantaneous tactile push button, to configure the Arduino's internal pull-up resistor architecture, and to implement software-based debouncing techniques.

---

### Technical Context: Internal Pull-Up Resistors & Debouncing

### The Problem of Floating Pins

When a microcontroller digital input pin is left disconnected from any voltage line (unpressed button state), it enters a **"floating" state**. In this state, atmospheric electrical noise causes the pin voltage to randomly oscillate between `HIGH` and `LOW`, generating erratic software readings.

### The Solution: `INPUT_PULLUP`

By declaring `pinMode(2, INPUT_PULLUP);`, we activate an internal resistor inside the ATmega328P chip connected to $+5\text{V}$. This pulls the pin default state securely to `HIGH`. When the button is pressed, it bridges the pin straight to Ground (`GND`), forcing a reliable, noise-free drop to `LOW`.

> **Inverted Logic Rule:** Because of the pull-up configuration, the input values operate on inverted logic:
> * Button **Unpressed** = `HIGH` (1)
> * Button **Pressed** = `LOW` (0)
> 
> 

---

### Pin Mapping Table

| From Arduino Uno Pin | To Breadboard / Button Pin | Component Function | Default Electrical State |
| --- | --- | --- | --- |
| Digital Pin 2 | Button Terminal (Top Pair) | Tactile Input Signal Line | `HIGH` (via $20\text{k}\Omega\text{--}50\text{k}\Omega$ internal pull-up) |
| GND | Button Terminal (Bottom Pair) | Ground Return Path | `LOW` ($0\text{V}$) |

---

### Software Implementation

Two methodologies were reviewed and tested to evaluate physical state changes:

### Approach A: Native Register Reading (`digitalRead`)

The default method uses standard Arduino functions to check the input state manually:

```cpp
void setup() {
  pinMode(2, INPUT_PULLUP); // Safe default state declaration
}
void loop() {
  int buttonState = digitalRead(2);
  if (buttonState == LOW) { /* Button is being pressed */ }
}

```

### Approach B: Hardware Abstraction Layer (`Button.h`)

To streamline development, an external button handling library (`Button.h`) was imported. This library automates the underlying register masks and abstracts state tracking.

---

### Observations

* **The Reality of Contact Bounce (Debouncing):** When writing code using raw `digitalRead()`, pressing the button once occasionally caused the code to act as if it were pressed 3 or 4 times. This is caused by "contact bounce"—microscopic, metallic vibrations occurring within the first few milliseconds of physical impact. Using the `Button.h` library instantly stabilized this, filtering out the mechanical noise automatically.
* **Inverted Logic Validation:** Testing verified that when the system reads a logic `0` (`LOW`), the user is actively pushing the button. Conversely, a logic `1` (`HIGH`) confirms an idle button.
* **Breadboard Orientation Errors:** An initial test placement rotated the button $90^\circ$ degrees out of true. This connected the input pin directly to Ground across the permanent internal pair, causing the program to register a permanent, un-releasable "pressed" state until the orientation was corrected.

---

### Conclusion

This sub-circuit introduced manual user interaction to our modular system. By taking advantage of the ATmega328P microcontroller's built-in internal pull-up resistors, we eliminated the need to insert external hardware pull-up or pull-down resistors into our breadboard matrix. Additionally, utilizing the `Button.h` library proved the vital role abstraction plays in mitigating real-world physical mechanical noise (debouncing) in embedded systems programming.

## Final Project – Integrated Smart Alarm Clock with Logic Disarm

### Objective
Synthesize the individual output (Buzzer, I2C LCD) and input (RTC, Push Button) sub-circuits into a unified, standalone system. To exceed baseline performance requirements, the system was engineered with an integrated **Mathematical Problem-Solving Disarm Mechanism** that forces cognitive engagement before silencing the alarm.

---

### System Architecture & Component Fusion

The final project seamlessly integrates all previous modules onto a shared breadboard layout using a centralized Arduino Uno microcontroller.

```
                          +-------------+
                          | Arduino Uno |
                          +------+------+
                                 |
        +-----------------+------+-----------------+-----------------+
        | (I2C Bus: A4/A5)|      | (Digital Pin 12)|      | (Digital Pin 2)
        v                 v      v                 v      v                 v
+-------+-------+ +-------+------+ +-------+-------+ +-------+-------+
|  I2C 2x16 LCD | |   I2C RTC    | | Piezo Buzzer  | |  Push Button  |
|  (Disp. Time) | | (Keep Time)  | | (Audio Alarm) | | (User Input)  |
+---------------+ +--------------+ +---------------+ +---------------+

```

---

###  Advanced Firmware Innovation: Mathematical Disarm

The system rejects standard, passive button-press silencing. When the current time matches the programmed alarm threshold, the system enters an active interrupt state:

1. **State Trigger:** The buzzer triggers a high-frequency rhythmic alarm tone.
2. **Dynamic Challenge Generation:** The LCD clears the time and dynamically displays a randomly generated arithmetic problem (e.g., $14 + 27 = ?$).
3. **User Interaction:** The user must utilize the push-button matrix to increment and submit the correct value.
4. **State Resolution:** The buzzer is silenced *only* when the calculated internal variable equals the user-submitted input.

---

### Final System Code Structure (`Smart_Alarm_Clock.ino`)

```cpp
#include <Wire.h>
#include <LiquidCrystal_I2C.h>
#include <RTClib.h>

// Component Initializations
LiquidCrystal_I2C lcd(0x27, 16, 2);
RTC_DS1307 rtc;

const int buzzerPin = 12;
const int buttonPin = 2;

// Alarm Variables
int alarmHour = 7;
int alarmMinute = 30;
bool isAlarmActive = false;

// Math Challenge Variables
int num1, num2, correctSolution;
int userAnswer = 0;

void generateMathProblem() {
  randomSeed(analogRead(A0)); // Seed random generator with floating noise
  num1 = random(5, 20);       // Generate random integer 1
  num2 = random(5, 20);       // Generate random integer 2
  correctSolution = num1 + num2;
  userAnswer = 0;             // Reset current user attempt
}

void setup() {
  pinMode(buzzerPin, OUTPUT);
  pinMode(buttonPin, INPUT_PULLUP);
  
  lcd.init();
  lcd.backlight();
  rtc.begin();
  
  // Optional: Uncomment below to set time during staging compilation
  // rtc.adjust(DateTime(F(__DATE__), F(__TIME__)));
}

void loop() {
  DateTime now = rtc.now();

  if (!isAlarmActive) {
    // Normal Operation: Display current time
    lcd.setCursor(0, 0);
    lcd.print(now.timestamp(DateTime::TIMESTAMP_TIME));
    
    // Check Alarm Condition
    if (now.hour() == alarmHour && now.minute() == alarmMinute && now.second() == 0) {
      isAlarmActive = true;
      generateMathProblem();
      lcd.clear();
    }
  } else {
    // Alarm Triggered State
    digitalWrite(buzzerPin, HIGH);
    delay(100);
    digitalWrite(buzzerPin, LOW); // Generate beeping sound

    // Display Puzzle on LCD
    lcd.setCursor(0, 0);
    lcd.print("Solve to Stop!");
    lcd.setCursor(0, 1);
    lcd.print(String(num1) + " + " + String(num2) + " = " + String(userAnswer));

    // Handle Button Input (Basic short press increments answer)
    if (digitalRead(buttonPin) == LOW) {
      delay(200); // Simple debounce delay
      userAnswer++;
      
      // Submit and verify if button is held, or if answer exceeds max threshold
      if (userAnswer == correctSolution) {
        isAlarmActive = false;
        digitalWrite(buzzerPin, LOW);
        lcd.clear();
        lcd.print("Correct! Morning!");
        delay(2000);
        lcd.clear();
      }
    }
  }
  delay(100);
}

```

---

### Observations

### A. Bus Load and Timing Latency

* **Observation:** Running real-time execution tracking alongside heavy display printing loops caused the buzzer beep cadence to stutter initially.
* **Technical Analysis:** Grouping the $I^2C$ transactions dynamically inside an active conditional check (`if (!isAlarmActive)`) freed up valuable clock cycles for the CPU to handle execution tasks while processing the active state array.

### B. Math Validation Logic

* **Observation:** During testing, the system sometimes initialized with identical math puzzles upon hard power cycles.
* **Technical Fix:** Implemented a `randomSeed()` pull driven by reading floating analog white noise on disconnected pin **`A0`**. This ensures unique mathematical equation generation every time the alarm activates.

### C. UI Limitations & Edge Cases

* **Observation:** Incremental entry using a single push button can prove tedious for high totals.
* **Design Consideration:** The puzzle generator ranges are strictly limited to lower boundaries ($5 \le \text{Value} \le 20$), ensuring solutions fall reliably between $10$ and $40$. This balances genuine cognitive engagement with efficient tactile input execution.

---

###  Conclusion

This final project showcases a robust integration of hardware components into a practical embedded system. By transitioning away from basic baseline design paradigms and incorporating custom random-variable math validation logic, the system effectively meets advanced project specifications. The resulting prototype serves as a strong portfolio example of responsive physical computing.
